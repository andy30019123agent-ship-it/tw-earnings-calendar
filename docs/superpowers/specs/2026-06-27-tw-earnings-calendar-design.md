# 台股法說會／財報行事曆＋分析工具 — 設計文件

> 日期：2026-06-27（台北）
> 狀態：設計待 Andy 過目
> 方案：A（獨立小工具＋週排程；研究段由 CC 對話即時做）

## 1. 目標（一句話）

每週六中午 12:00 自動掃描全市場，預告「下一週」有法說會／財報的公司並推播給 Andy；Andy 圈選標的後，由 CC 即時上網做財報＋法說會分析（先快報、可展開完整版），並把每份研究報告存進一個公開的「研究存放網頁」長期保存、可隨時翻閱。

## 2. 使用者與情境

- 使用者：Andy（非工程師，主要在 Discord、次要 Telegram 用手機看）。
- 痛點：想提前知道在意的股票何時開法說會／公布財報，並在當下拿到重點分析，但不想自己一家一家查、也不想浪費力氣研究自己沒興趣的公司。
- 成功標準：
  1. 每週六中午能在手機收到一張「下週行程表」，清楚、好挑。
  2. 回一句代號就能拿到該股快報；想深看再展開完整版。
  3. 過去做過的研究報告都收在一個網頁，之後想回顧某檔看得到。
  4. 全程不需要他開電腦、不需要技術操作。

## 3. 系統概觀

兩段式，刻意把「掃描（自動、程式做）」與「研究（即時、CC 做）」分離：

```
第一段：每週六 12:00（台北）  ── GitHub Actions cron
  掃 MOPS 行事曆 → 篩「下週一～下週日」有法說會/財報的公司
  → 取市值排序 → 組成時間表(JSON + 訊息文字)
  → 推 Telegram（bot sendMessage）＋ Discord（webhook）

第二段：研究（事件「前」快報 → 事件「後」詳細） ── 活著的 CC 對話
  Andy 回「研究 2330 2454」
    → CC 即時產【事前快報】(近期業績/題材/多空/隱藏機會/後勢預測) → 推回 + 上架
    → 記進 picks.json（含事件日）
  每日排程：picks 中「昨天開完法說會/財報」的 → 自動推提醒「回我『詳細 2330』」
  Andy 回「詳細 2330」
    → CC 即時查事件實際內容，產【詳細報告】(現況/題材機會與風險/多空/隱藏題材/隱藏合作機會)
    → commit/push → 觸發網站重建

第三段：研究存放網頁 ── GitHub Pages（公開連結）
  把所有研究報告（快報＋詳細）收成可翻閱的清單（依股票／日期）
  首頁同時顯示最新一期行事曆
```

關鍵原則：**研究「怎麼做」不寫進程式碼**——研究是 CC 即時做的事（會用 WebSearch／WebFetch），程式只負責「定時把行程表推出來」＋「把 CC 產出的報告檔案渲染成網頁」。這樣工具本體很小、好維護，研究品質也不被死碼綁住。

## 4. 第一段：掃描與推播（程式）

### 4.1 時間窗

- 在「週六」執行，掃描範圍 = **下週一 00:00 ～ 下週日 23:59（台北）**。
- 以執行當天為基準計算下週一日期，不寫死。

### 4.2 資料來源（全部免費官方源）

| 內容 | 來源 | 備註 |
|------|------|------|
| 法說會預告 | MOPS 公開資訊觀測站「法人說明會一覽表」 | 公司事前申報，較可靠 |
| 財報／董事會預告 | MOPS「重大行事曆／董事會召開預告」（含議程「承認財務報告」者） | 用「預告的董事會日」近似財報公布日 |
| 市值排序 | TWSE／TPEX：收盤價 × 已發行股數 | 股數表用 TWSE/TPEX 公司基本資料，低頻更新可快取 |

> 實作前須先以小腳本驗證 MOPS 各 endpoint 的實際網址與回傳格式（MOPS 介面偶有改版），確認後再寫正式爬取。寫入計畫的第一個 Task 就是「驗證資料源」。

### 4.3 市值取得策略

- 主：建立「代號 → 已發行股數」快取表（TWSE 上市、TPEX 上櫃公司基本資料），市值 = 最新收盤價 × 股數。
- 備援：若某股抓不到股數，改用「成交值（收盤價 × 當日成交量）」當熱門度近似排序，並在該筆標註「(以成交值估)」。
- 股數表每月或每季更新一次即可（變動低頻）。

### 4.4 輸出格式

時間表「每天一組、組內市值大→小」，每筆欄位：

```
[日期 星期]
  代號 名稱   類型(法說會/財報)   產業   市值(億)
```

範例（推播文字）：

```
📅 下週法說會／財報預告（6/29 ~ 7/5）

▍7/1（週三）
  2330 台積電   法說會   半導體   28,500 億
  2454 聯發科   法說會   半導體    9,200 億
  ...

▍7/3（週五）
  2317 鴻海   財報(董事會)   電子   7,800 億
  ...

回我代號就幫你研究，例如：研究 2330 2454
```

- 同時產出機器可讀 `latest.json`（供研究存放網頁首頁顯示最新行事曆）。
- 推播管道：
  - **Telegram**：推到 Andy 的私人助手 bot「金冬天秘書」@winteragent_bot（chat id `542348223`）。bot token 放 GitHub Secrets，永不進版控。
  - **Discord**：Discord webhook（同一份內容兩邊都發，Andy 兩邊都看得到）。

### 4.5 邊界與失敗處理

- **下週沒有任何法說會／財報**：照樣推一則「下週暫無已申報的法說會／財報行程」，不要靜默（避免讓人以為壞了）。
- **MOPS 抓取失敗／逾時**：重試（含 curl／urllib 雙路徑），全失敗則推「行事曆抓取失敗，稍後人工補」並讓 Actions 標記失敗，不要推出空表假裝成功。
- **財報實際公布日不可得**：源頭限制——台股財報實際公布日不一定事先公告，本工具以「公司在 MOPS 預告的董事會／法說會日」為準；少數未預告者可能漏列。這點在推播表尾用一行小字說明，誠實標示。

### 4.6 排程

- **每週掃描**：GitHub Actions cron `0 4 * * 6`（UTC 週六 04:00＝台北週六 12:00）→ 掃下週行事曆並推播。
- **每日提醒**：GitHub Actions cron `0 3 * * *`（UTC 每日 03:00＝台北每日 11:00）→ 讀 `picks.json`，把「昨天有開法說會／財報」的追蹤標的推提醒（見 5.3）。
- 金鑰（TG bot token／chat id、Discord webhook URL）只放 GitHub Secrets，永不進版控。
- 兩個排程都支援 workflow_dispatch 手動觸發以便測試。

## 5. 第二段：研究（CC 即時做；事前快報 → 事件後自動提醒 → 事後詳細）

研究分兩個時間點，對應事件的「前」與「後」：

### 5.1 觸發與追蹤狀態

- 週六行事曆推出後，Andy 回「研究 2330 2454」。
- CC 收到後做三件事：
  1. 立刻產出每檔的**事前快報**（見 5.2），推回 Discord／TG。
  2. 把這些標的記進 `data/picks.json`：`{代號: {name, event_date, event_type(法說會/財報), status:"watching"}}`，event_date 從當期 `latest.json` 對應取得。
  3. commit/push（picks.json ＋ 快報 md），供每日提醒排程讀取、並把快報上架網頁。
- 指令格式寬鬆，CC 需能理解：「研究 2330 2454」「詳細 2330」「2330 先別放」等。

### 5.2 事前快報（挑完立刻給；事件「前」的預判）

每檔涵蓋（重點在「評估與預測」，不只羅列數字）：

- **近期業績**（最近一季 EPS／營收 YoY‧QoQ、毛利率方向）
- **近期題材＋可行性評估**：題材是什麼，並判斷**可不可行、落地機率與時程、風險在哪**（真題材 vs 炒作）
- **多空評估**（偏多／偏空／中性＋理由）
- **隱藏題材／隱藏合作機會**（市場較少注意到的潛在題材、可能的客戶／供應鏈／策略合作線索）
- **成長機會**（未來成長空間、驅動力、可放大的點）
- **後勢預測**（事件前的預判：可能怎麼走、要盯什麼觀察點）

基於「目前已知資訊」（最近財報、新聞、題材），明確標示這是事件前預判。

### 5.3 事件後自動提醒（每日排程，方案 1）

- 每天早上 GitHub Actions cron 讀 `picks.json`，找出「event_date == 昨天」且 status=watching 的標的。
- 命中 → 自動推一則到 TG＋Discord：「📣 2330 昨天開完法說會了，回我『詳細 2330』就幫你出完整報告」。
- 提醒以「預告的事件日」為準（財報實際日不可得的源頭限制照舊適用）；提醒推出後把 status 標為 `event_passed`。

### 5.4 事後詳細報告（Andy 回「詳細 2330」→ CC 即時做）

CC 即時上網查證事件實際內容（法說會簡報／Q&A、實際財報數字、後續新聞），產出單檔小報告。**核心是評估與預測**，不只覆述法說會內容：

- **現況**（最新財報實際數字、營運近況）
- **題材可行性與風險評估**：逐個題材判斷可不可行、落地機率與時程、潛在風險（真題材 vs 話術）
- **多空評估**（綜合判斷＋理由）
- **隱藏題材**（市場尚未充分反映、法說會字裡行間透露的點）
- **隱藏合作機會**（潛在客戶／供應鏈／策略合作線索）
- **成長機會與後勢預測**（未來成長空間、驅動力、預期走勢與觀察點）
- 來源連結彙整於文末（沿用 gooaye-digest：內文不插連結，末尾列來源）

> 撰寫立場：當一個會質疑的分析師，不照單全收公司說法；對每個題材給「我的判斷」而非只轉述。判斷需有依據，無依據處明講「待觀察」。

完成後上架網頁、標記「詳細報告＋日期」，並把該股 picks status 標為 `done`。

### 5.5 品質紀律與優先任務

- **優先主動挖掘「隱藏題材與機會」**：報告最有價值的部分是市場還沒充分注意到的點——潛在題材、潛在合作（客戶／供應鏈／策略）、被低估的成長機會。每份報告都要花最多力氣在這塊，而非複述公開新聞。
- 數字務必查證、標日期，寧缺勿錯（沿用儀表板的新聞紀律）。
- 查不到的明講「查無」，不杜撰；事前快報與事後報告都標清楚「資料時間點」。

## 6. 第三段：研究存放網頁（GitHub Pages，公開連結）

### 6.1 用途與隱私

- 一個公開連結的靜態網站（Vite + React，跟選股站同款），把 CC 做過的研究報告長期存起來、可翻閱。
- 隱私：採**公開連結**（Andy 已確認可接受）。內容是公開股票的分析，不含個資；網頁尾標「僅供研究參考，非投資建議」。

### 6.2 報告怎麼上網（發佈流程）

- CC 做完研究後，把報告寫成檔案存進 `web/reports/<代號>-<日期>.md`（含 frontmatter：代號、名稱、日期、深度 快報/完整）。
- CC commit/push 到 main → GitHub Actions 自動 build + 部署 → 網頁更新。
- 預設「做完就上架」；若 Andy 對某篇說「先別放」，該篇就不寫檔。
- 「事前快報」與「事後詳細報告」**都保留**（同股的兩篇互補：事前預判 vs 事後驗證），檔名以「代號-類型-事件日」為鍵；同類型同事件日重做才覆蓋。

### 6.3 網頁內容

- **首頁**：最新一期行事曆（讀 `latest.json`）＋ 最近更新的研究報告列表。
- **報告庫**：所有報告依「股票」或「日期」瀏覽、可搜尋代號／名稱；點開讀全文（markdown 渲染）。
- 純前端、無後端、無登入；報告即 repo 內的 `.md` 檔，網站只負責渲染。

### 6.4 邊界

- 報告檔不存在時頁面顯示空狀態，不報錯。
- markdown 內容由 CC 產出，發佈前 CC 自審（沿用品質紀律）。

## 7. 專案結構（方案 A）

新獨立 repo：`tw-earnings-calendar`（推到 andy30019123agent-ship-it）。

```
tw-earnings-calendar/
  scripts/                # 第一段＆提醒：掃描／推播（Python）
    fetch_calendar.py     # 掃 MOPS 法說會＋董事會行事曆，篩下週
    market_cap.py         # 股數快取表 + 市值計算（備援成交值）
    build_message.py      # 組推播文字 + latest.json
    push.py               # 推 Telegram + Discord webhook（共用）
    run_weekly.py         # 週六入口：掃描→推行事曆
    run_daily_remind.py   # 每日入口：讀 picks.json→推事件後提醒
    lib/                  # 共用：日期、HTTP 重試、parser
  web/                    # 第三段：研究存放網頁（Vite + React）
    src/                  # 首頁(行事曆+最新報告) + 報告庫 + 報告閱讀頁
    reports/              # CC 產出的研究報告 .md（含 frontmatter：代號/名稱/日期/類型 快報|詳細）
    public/data/          # build 時放入 latest.json 供首頁讀取
  data/
    shares_cache.json     # 代號→股數（低頻更新）
    latest.json           # 最新一次行程表（同步給 web）
    picks.json            # Andy 追蹤中的標的＋事件日＋狀態（CC 寫、提醒排程讀）
  tests/                  # pytest：日期窗、parser、picks 篩選、空表/失敗處理
  .github/workflows/
    weekly.yml            # 週六 cron：掃描→推行事曆→更新 latest.json→重建站
    daily.yml             # 每日 cron：讀 picks.json→推事件後提醒
    deploy.yml            # push 到 main（新增報告/picks）→ build + 部署 Pages
  docs/superpowers/...
  README.md               # 記「研究指令格式」備忘，方便任何 CC session 接手
```

研究「怎麼做」不寫成程式——它是 CC 收到代號時即時做的對話流程；程式只負責把 CC 產出的報告 `.md` 渲染成網頁。

## 8. 不做（YAGNI）

- 不做後端／會員登入／資料庫（網頁是純前端，報告即 repo 內 md 檔）。
- 不做個股長期數據庫／回測。
- 不做自動投資建議或下單。
- 不把研究「怎麼做」寫成程式（保持由 CC 即時做，品質與彈性較好）。
- 不接付費資料源（全用免費官方源）。
