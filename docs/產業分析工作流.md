# 產業分析報告工作流（唯一權威 SOP）

> **本檔是 tw-earnings-calendar「產業分析報告」的唯一權威 SOP。**
> 任何有能力的模型只要拿到「這份文件 ＋ 一個產業名稱」，就要能獨立產出一份產業分析報告，放進 `web/reports/`，品質對齊個股報告。
> 個股報告的 SOP 是 [`研究報告工作流.md`](研究報告工作流.md)；本檔是它的「產業層級」姊妹版，**共用的核心紀律（準確第一、標記系統、會質疑、Bear case、scorecard 信心校準）完全沿用**，本檔只寫「產業層級才有」的差異。衝突時本檔對產業報告為準。
> 服務對象：散戶投資人 Andy（非工程師）。他要的核心問題是：**「這個產業現在值不值得投資？如果要投，選誰比較好？」** 白話、少術語、每個數字翻成投資意義。

---

## 0. 使命與立場

### 0.1 你在回答的一個問題
產業報告從頭到尾在回答一件事：**「這個產業值不值得投資、要投選誰。」** 前面所有章節（產業現況、產業鏈、龍頭比較、股價籌碼）都是為了支撐最後的「投資評斷 ＋ 選誰」。不要寫成產業百科。

### 0.2 立場＝會質疑的產業分析師
- **禁止**替產業/公司護航、複述新聞或法人的樂觀說法。族群性狂熱（大家一起漲）要點破「這是 beta（整個族群被行情帶）還是 alpha（這家真的比較強）」。
- **優先挖**：市場還沒充分反應的產業轉折、被低估的二階受惠、被高估的「本夢比」個股。
- **Bear case 強制**：每份報告要有「站在看空這個產業的人立場，最強的理由是什麼」，不可省略、不可敷衍。

### 0.3 🔴 最高原則：準確第一（違反即作廢）
- 寧可少講、不要講錯；完整度讓位給正確性。
- **每個數字標來源＋資料日期**；查不到就寫「查無/待確認」，不腦補、不用印象填空。
- **標記系統**（報告開頭 blockquote 必聲明、正文逐句用）：`✅`＝已驗證事實(附來源/日期)｜`🔮`＝推測/預判｜`🟡`＝待確認/有時間差(即時股價、當日籌碼)｜`(估)`/`約`＝推估值要說依據。
- 個股層級的數字（單一公司毛利、EPS、股價位置）一律以 [`研究報告工作流.md`](研究報告工作流.md) 的資料源與算法為準，不在本報告重新憑記憶編。

## 1. 觸發、檔名、frontmatter

- **觸發語**：Andy 說「產業分析 <產業>」（例：產業分析 功率半導體）。一次可多個產業＝並行各出一份。
- **檔名格式**（寫死）：`web/reports/<產業>-產業分析-<YYYY-MM-DD>.md`
  - `<產業>` 用中文產業/題材名（例：功率半導體、封測、CoWoS）。`<YYYY-MM-DD>`＝報告產出日，動手前先 `TZ=Asia/Taipei date "+%Y-%m-%d"`。
  - 前端 `web/src/lib/reports.js` 的 FILENAME_RE 已支援 `產業分析` 類型；id 群組吃中文產業名。
- **frontmatter 五欄**（缺一不可）：
  ```yaml
  ---
  id: 功率半導體          # 產業名（當唯一鍵，非股票代號）
  name: 功率半導體         # 顯示名
  date: 2026-07-06        # 報告產出日
  type: 產業分析           # 固定字串，前端分類靠它
  event_date: 2026-07-06  # 產業報告無單一事件，填產出日或該產業最近關鍵時點
  ---
  ```

## 2. 成分股界定（產業報告獨有的第一步）

1. **先圈成分股**：用你的產業知識＋查證，列出台股裡屬於這個產業的上市櫃公司（含代號、上市.TW/上櫃.TWO）。寧可多列幾檔候選，再篩。
2. **Andy 可加減**：報告開頭要明列「我納入了哪些、為什麼；哪些邊緣案例排除、為什麼」，讓 Andy 一眼能說「加某檔、拿掉某檔」。
3. **分層**：挑 **3~5 檔龍頭/代表**做深入分析（第 3.3 節），其餘成分股列進比較表（3.4）帶過即可，不必每檔深挖。
4. 每檔的市場別（.TW/.TWO）一律實測 Yahoo chart API 確認，不憑記憶（個股 SOP 2.4）。

## 3. 報告章節規格（8 段）

> 主軸：**現況（1~5 段）→ 判斷（6~8 段）**。前五段建立事實，後三段給評斷與選擇。

### 3.1 產業現況＋景氣位置
- 這個產業現在在景氣循環的哪個位置？供需鬆緊、漲價/跌價循環、稼動率、庫存。是結構成長還是循環反彈？（點破「別把循環當結構」）
- 帶出這個產業當下最大的題材（AI、車用、缺貨漲價…）與它的可驗證程度。

### 3.2 產業鏈地圖
- 上中下游怎麼串；這個產業卡在哪個位置（越上游/越關鍵＝議價權越高）。
- **二階/三階受惠**：賣鏟子給這個產業的人。**同機櫃/同訂單鄰居**：可交叉驗證需求動能。

### 3.3 龍頭公司深入分析（3~5 檔）
- 每檔一小段：做什麼、在產業鏈的位置（Tier-1 直供 or Tier-2）、最新一季硬數字（營收/EPS/毛利，標日期）、獨特優勢或罩門。
- 個股數字引用個股 SOP 的資料源；已有個股報告的檔要連結、延續其非共識線索。

### 3.4 各家差異比較表
- 一張表把所有成分股放進來比：股價/PE/PB/近一季 EPS 成長/毛利率/題材純度/技術路線/市場別。
- **表後必附 `> 🔮 解讀：`**，點出這組在賭什麼「分歧」（例：高壓 vs 中低壓、買現在的獲利 vs 買轉型夢）。表是骨、解讀是肉。

### 3.5 股價＋籌碼
- 各龍頭股價在一年區間的**位置百分位**、近半年漲跌幅（判斷是否已非低基期、追高風報比）。
- 三大法人族群動向、有無「法人 vs 散戶背離」、處置/注意股。撈不到就標查無，不編。

### 3.6 產業投資評斷（值不值得進場）
- 綜合前五段，給一句話定調：這個產業現在**值不值得投資**、理由、以及「什麼條件成立才進場、什麼訊號出現要退場」。
- 明說現在買的是「便宜的獲利」還是「貴的夢」。

### 3.7 選誰（穩健型 vs 想衝型 ＋ 綜合排序）— 本報告的重點交付
- **穩健型適合誰**：獲利實在、估值不極端、下檔有撐的（附理由）。
- **想衝型適合誰**：題材純度高、彈性大但風險高的（附理由）。
- **綜合排序**：把成分股排出第一、第二…並各給一句「贏在哪、輸在哪」。
- 每個「選誰」的判斷旁標 **信心 X/5 ＋一句理由**（見第 6 節），並登錄 scorecard。
- 收尾用「雙路線」語氣，尊重不同風險偏好，不替 Andy 下單一結論。

### 3.8 空方風險（Bear case）
- 站在看空整個產業的人立場，最強的理由（需求證偽、漲價循環反轉、產能過剩、地緣/政策、估值泡沫）。要有殺傷力，不是風險提醒貼紙。

### 3.9 文末：資料時間點 ＋ 來源彙整 ＋ 免責
- `資料時間點`：哪些是已公布財報、哪些法人估、哪些推測、股價哪天。
- `來源彙整`：每則具體標題＋日期＋連結。
- 固定免責：`> 本報告為 AI 輔助研究，僅供參考，不構成投資建議。`

## 4. scorecard 產業預測登錄與回驗

- 沿用個股同一個 `web/src/data/scorecard.json`、同一套 schema。差異：
  - `stock_id`／`stock_name` 填**產業名**（例：`功率半導體`），或填該預測主要指向的個股代號（若預測是「首選 X 會跑贏族群」就用 X 的代號）。
  - `pid` 格式沿用：`<產業名或代號>-<predicted_on>-<兩位序號>`。
  - `report` 填本報告檔名。`category` 沿用五類。`confidence` 1~5 必填。
- **可驗證的產業預測**例：「族群 H2 景氣續揚」「首選 X 未來一季跑贏族群平均」「族群估值偏高、H2 有回檔壓力」。給 `direction/price_range/deadline`。
- **信心校準**：擇時/選股類本質難準，別每筆都給高分。目標是「越有把握＝越常命中」，未來可回驗哪類判斷可信。
- 回驗規則同個股 SOP 第 5 節：寫新報告時順手回驗同產業舊預測；每月全面回驗到期的。

## 5. 執行 checklist

1. `TZ=Asia/Taipei date` 確認台北日期。
2. 圈成分股（第 2 節）→ 分龍頭/其餘。
3. 蒐集：產業供需/漲價/題材（近 3 個月）→ 各龍頭最新財報數字 → 股價位置/籌碼。第一手優先、撈不到標查無。
4. 按 8 段寫；每個數字配解讀；標記系統逐句用對；比較表後附解讀；Bear case 要硬。
5. 自我審查（全過才交付）：成分股界定有沒有說明納入/排除？比較表後有沒有解讀？投資評斷有沒有給進場/退場條件？選誰有沒有「穩健+想衝+排序+信心」？空方夠不夠強？每個數字有來源日期？查不到誠實寫查無？frontmatter 五欄齊、檔名對、日期是今天？
6. 寫檔 → scorecard 登錄 → build → commit push → 線上驗證 → 發連結。

## 6. 前端接線與上架（架構備忘）

- **報告類型**：`web/src/lib/reportTypes.js` 需含 `產業分析` 條目（label/description/color/badgeClass）並加進 `REPORT_TYPE_ORDER`；`ReportLibrary.jsx` 篩選器自動吃。
- **檔名解析**：`web/src/lib/reports.js` 的 `FILENAME_RE` 的 type 群組需含 `產業分析`。
- **badge 樣式**：`web/src/index.css`（或 App.css）比照 `badge-flash` 等加 `badge-industry` 顏色。
- **上架**：`git add web/reports/ web/src/data/scorecard.json` → commit → push main，`deploy.yml` 自動 build 上架 GitHub Pages，線上驗證 `r/<檔名去掉.md>/` 可開（2026-07-07 起報告改真實路徑，發連結一律用此格式才有正確分享預覽卡；舊 `#/r/` 僅自動轉址用）。

## 7. 調度規範（主對話/指揮者）

- 每個產業派一個 **Opus（或當時最高階）** subagent，強制第一步＝完整讀本檔 ＋ 個股 SOP 的冷啟動範例。
- 多產業並行時各 agent 只寫自己的 `web/reports/<產業>-產業分析-<日期>.md`；`scorecard.json` 只讀不寫，預測回報給主對話統一登錄（避免並行互蓋）。
- agent 回報格式：檔案路徑／一句話投資評斷／成分股清單（含納入排除理由）／選誰排序＋信心／新預測清單（claim・direction・price_range・deadline・category・confidence＋理由）。
- 主對話統一登錄 scorecard、commit push、驗線上、發 Andy。品質守門：抽查成分股界定、比較表解讀、投資評斷、信心是否誠實。
