# 研究報告工作流（唯一權威 SOP）

> **本檔案是 tw-earnings-calendar 研究報告的唯一權威 SOP。**
> 任何模型（Claude Opus 或其他有能力的 LLM）只要拿到「這份文件 ＋ 一個股票代號」，就要能獨立產出與 `web/reports/` 現有報告同等級的研究報告。
> 當本檔與 README、記憶、印象衝突時，**以本檔為準**。本檔更新時要同步修正 README 對應段落（但你在單次任務中若被交代「只寫這份、別動 README」，就照交代做）。
>
> 服務對象：散戶投資人 Andy（非工程師）。他要的是「快速掌握公司**現況＋未來**、作為投資參考」，核心是**評估與預測**，不是抄一堆數字。

## 0. 使命與立場

### 0.1 你在為誰、為什麼而寫
- 讀者是**散戶投資人 Andy**，不是機構分析師、不是工程師。用白話、少術語；術語第一次出現要順帶一句解釋。
- 報告目的＝讓 Andy 在幾分鐘內掌握「**這家公司現在怎麼樣、未來會怎麼樣**」，作為投資判斷的參考依據。
- **核心是評估與預測，不是資料搬運。** 每個數字後面都要接「所以呢？」——把數字翻成投資意義。裸數字沒有價值。

### 0.2 你的角色＝「會質疑的分析師」
- **禁止**替公司護航、複述公司/新聞的樂觀說法。公司說的每句話都要問「這是財測還是喊話？可驗證嗎？」
- **優先挖**市場還沒充分反應的隱藏題材、被低估的機會、被輕描淡寫的結構風險。
- **認真寫空方（Bear case）**：每份報告都要有一段「站在放空這檔的人立場、最強的看空理由是什麼」。這是對抗偏多偏誤的強制機制，不可省略、不可敷衍。

### 0.3 🔴 最高原則：準確第一（違反此條，整份報告作廢）
- **寧可少講，不要講錯。** 完整度永遠讓位給正確性。
- **每個數字都要標來源＋資料日期。** 沒有來源、沒有日期的數字不准出現。
- 查不到 → 直接寫「**查無**」或「待確認」，不要用印象填空、不要腦補。
- **標記系統**（每份報告開頭的 blockquote 必須聲明這套圖例，正文逐句使用）：
  - `✅` ＝已驗證事實（後面必附來源／日期）
  - `🔮` ＝推測／預判（你的判斷，不是事實）
  - `🟡` ＝待確認／數據可能有時間差（例：即時股價、當日籌碼）
  - `(估)` `約` ＝推估值，要說明推估依據
- **絕不把猜測當事實陳述。** 低信心的判斷要明講「這是推測／待證實」。原則是「可以不確定，但一定要提示不確定」——讓 Andy 知道哪些能信、哪些待查。
- 若事後發現先前報告判斷有誤，**開頭顯著更正**（見 [附錄 A 規則 13](#附錄-a從範例報告歸納的隱性品質規則)），不要偷偷改掉。

## 1. 報告類型與觸發時機

`type` 只有三種，報告庫篩選器依此分類，不得自創：

| type | 觸發 | 時機 | 說明 |
|---|---|---|---|
| `快報` | Andy 說「研究 <代號>」 | 法說會/財報**事前** | 事前預判：近期業績＋題材＋多空＋隱藏題材 |
| `法說會` | Andy 說「詳細 <代號>」 | 法說會**事後** | 以官方簡報＋報導查證，含預測追蹤表 |
| `財報` | Andy 說「詳細 <代號>」（該事件是財報） | 財報**事後** | 同上，事件是財報發布 |

> 舊的 `詳細` 一律視為 `法說會`。新報告只用上述三類之一。

**事後報告的時機鐵則**：以法說「**隔天**」產出為原則。當天官方簡報常還沒上傳 MOPS、媒體報導也不齊；隔天內容才齊。若被要求當天就寫，要在誠實聲明裡標「官方簡報尚未上傳，本文依媒體報導＋既有財報」。

**檔名格式**（寫死，不可變）：
```
web/reports/<id>-<類型>-<YYYY-MM-DD>.md
例：web/reports/3526-快報-2026-06-27.md
    web/reports/3526-法說會-2026-07-02.md
```
`<YYYY-MM-DD>` ＝**報告產出日**（不是事件日）。動手前先跑 `TZ=Asia/Taipei date "+%Y-%m-%d"` 確認台北日期。

**frontmatter 格式**（每份報告最上方，五個欄位缺一不可）：
```yaml
---
id: 3526
name: 凡甲
date: 2026-06-27        # 報告產出日
type: 快報              # 快報 / 法說會 / 財報 三選一
event_date: 2026-07-01  # 對應的法說會或財報日期
---
```

## 2. 資料源作戰地圖

**每個來源都要問三件事：查什麼、怎麼拿、拿不到怎麼辦。** 拿不到就標「查無」或換替代源，**禁止**用記憶填補。

### 🚨 資料蒐集鐵律（違反＝報告不合格，直接退回重做）

> （2026-07-07）症狀：上詮 3363 事後報告交出後被 Andy 打回——籌碼面整段「查無」、只用二手新聞拼湊、官方法說簡報完全沒讀、股價數字還寫錯（631 vs 實際 603）。→ 根因：subagent 遇到「打不開的第一手材料」（官方簡報是掃描影像 PDF、一般讀取工具只回傳亂碼）就直接跳過標查無，把「沒去查」和「查不到」混為一談，用查無當免死金牌。→ 以下四條規則寫死：

1. **禁止「裸查無」**：標「查無／待確認」時，**必須附上「查了哪個來源＋為什麼失敗」**（例：「查了 Yahoo `institutional-trading`✅ 拿到法人、`margin-trading` 回 404、Goodinfo 頁面需 JS 空白，故融資融券未取得」）。沒有查詢軌跡的「查無」＝偷懶，不合格。同一項至少換 2 個來源再宣告查無。
2. **官方簡報是掃描影像 PDF 的讀法**（第一手鐵則，不可跳過）：WebFetch 讀 MOPS 簡報 PDF 若回傳亂碼／JPEG 流／`(QE QE…)` 二進位，**不准就此標查無**。改用：①WebFetch 會把 PDF 存到本地 `tool-results/`，用 `pymupdf(fitz)` 把每頁 render 成 PNG（`page.get_pixmap(dpi=120).save(...)`）②再用 Read 工具「看圖」讀出財務表、產品占比、量產時程。上詮案例證實：官方簡報藏著 Q1 完整三大表（毛利率崩到 2%）與 FAU 官方時程，這些二手新聞全沒有。（poppler 未裝、`brew` 不在背景 PATH；pymupdf 已裝可直接用。）
3. **資料源打卡表**：事後報告交付前，SOP 2.1~2.6 每個來源都要留下狀態——官方簡報／完整財務三表／月營收／估值(PB/PE/淨值)／籌碼(三大法人/融資融券/外資持股)／股價區間，逐項「查了沒、拿到什麼」。主對話品質守門時逐項對照，缺一項且無查詢軌跡就退回。
4. **數字必核，不沿用不憑印象**：股價、漲跌幅、區間、PB、市值一律用**當前原始資料**（Yahoo chart API、官方財報）重算，**不准沿用舊報告數字或憑印象**（上詮 PB 因 2 月現增從近 30 倍腰斬到約 12 倍，沿用舊值就會錯得離譜）。

### 2.1 MOPS 法說會官方簡報 PDF —— 第一手鐵則（事後報告必先做）
- **查什麼**：法說會的官方簡報原文（公司自己講的展望、財測、產品占比、市佔）。
- **怎麼拿**：法說會一覽表（見 2.2）的每一列有「簡報內容」PDF 連結與「影音連結」欄位。用 **WebFetch 直接讀 PDF 原文**。簡報 URL 常見型態：
  `https://mopsov.twse.com.tw/nas/STR/<代號><民國年月>M<序號>.pdf`
  （範例：力積電 6/29 簡報 `https://mopsov.twse.com.tw/nas/STR/677020260626M001.pdf`）
- **鐵則**：寫法說會事後報告，**先讀官方簡報原文，媒體報導只當補充**。
- **拿不到怎麼辦**：簡報尚未上傳很常見。明標「官方簡報尚未上傳，本文依據媒體報導＋既有財報」，並在誠實聲明裡寫清楚，日後可回補。

### 2.2 MOPS 法說會一覽表 API
- **查什麼**：某一年某市場的所有法說會場次（含日期、簡報 PDF、影音連結）。
- **怎麼拿**：`POST https://mopsov.twse.com.tw/mops/web/ajax_t100sb02_1`
  - 必帶 Header：`Referer: https://mopsov.twse.com.tw/mops/web/t100sb02_1`、正常 `User-Agent`（缺這兩個會被擋）
  - body（完整）：`encodeURIComponent=1&step=1&firstin=1&TYPEK=sii&year=115&month=`（`TYPEK=sii` 上市／`otc` 上櫃；`year=<民國年>`，例 2026 → `115`；`step/firstin` 缺了會拿不到資料）
  - 回傳 HTML 表格，解析邏輯見 `scripts/fetch_calendar.py`（民國日期轉換、去 tag、只留 4 位純數字代號）。
- **拿不到怎麼辦**：改用個股頁或搜尋「<公司> 法說會 簡報」找官方 PDF。

### 2.3 TWSE OpenAPI（結構化財務數字，第一手）
基底 `https://openapi.twse.com.tw/v1/`，回傳 JSON，全市場一次撈：

| endpoint | 內容 |
|---|---|
| `t187ap04_L` | 重大訊息 |
| `t187ap05_L` | 每月營收 |
| `t187ap06_L_*` | 損益表（依產業別分檔） |
| `t187ap07_L_*` | 資產負債表（依產業別分檔） |

- **注意**：上市用 TWSE、上櫃要找對應 TPEX 來源。撈到後自己算年增/季增，別直接抄新聞的成長率。
- **拿不到怎麼辦**：改用財報狗/Goodinfo 等公開頁交叉核對，並標資料日期。

### 2.4 即時股價／歷史價 —— Yahoo chart API
- **怎麼拿**：`https://query1.finance.yahoo.com/v8/finance/chart/<id>.TW`
  - 上市＝`.TW`，上櫃＝`.TWO`（搞錯會抓不到或抓錯檔）。
- **用途**：現價、一年區間高低、算「位置百分位」與「近半年漲跌幅」（事後報告判斷是否已非低基期，見附錄 A 規則 9）。
- **拿不到怎麼辦**：標 `🟡 股價待確認`，用最近一筆可考收盤價並註明日期。

### 2.5 估值與籌碼
- **估值**（PE/PB vs 歷史與同業）：網路搜尋財報狗、Goodinfo、玩股網等公開頁。PE 用**法人估 EPS** 算時要標 `🔮 EPS 為法人估、非公司財測`。
- **籌碼**（三大法人買賣超、外資持股、融資融券、處置/注意股）：TWSE／TPEX 公開資料或財經網站。
- **拿不到怎麼辦**：當日籌碼常撈不到——直接標「當日三大法人資料源暫缺（查無/待確認）」，不要編。範例見 `3526-法說會-2026-07-02.md` 的籌碼段。

### 2.6 新聞題材
- **查什麼**：近 **3 個月**內的題材、法說報導、合作案。來源：經濟日報、工商時報、鉅亨網、科技新報、財訊、TechNews 等。
- **鐵則**：每則都標日期；官方簡報優先於媒體；財訊/萬寶等「觀點文」當線索、不當事實。
- **拿不到怎麼辦**：標「查無近期題材新聞」，以既有財報為基礎做評估。

## 3. 報告章節規格

以下章節為「涵蓋標準」。**快報**與**事後（法說會/財報）**共用大骨架，差異標在各節。每節格式：`目的｜必答問題｜資料來源｜品質標準｜常見錯誤`。

> **「現況 vs 未來」主軸**：
> - 回答**現況**的章節：近期業績、估值面、籌碼面、財務品質紅旗、法說實際重點（事後）。
> - 回答**未來**的章節：催化劑時間軸、多空評估、深度延伸（供應鏈/情境/非共識）、後勢預測、預測追蹤（事後）。
> 每份報告都要「現況」與「未來」兩腳都站到，不能只寫現況（變成財報摘要）或只寫未來（變成作文）。

### 3.1 一句話定位／總結
- **目的**：一句話讓 Andy 知道這檔的本質，並點出「這決定了什麼」。
- **標題規範**：此段標題固定用 `## 一句話總結`（或 `## 一句話定位`），**每份報告必有、放在第一個 h2**——網頁的「重點速覽卡」靠這個標題抽取，漏了或改名，報告頁就不會顯示摘要卡。
- **必答**：公司做什麼？為什麼這是這次事件的看點？（事後）法說會真正的主軸是什麼？
- **品質標準**：不只說「做連接器」，要說「做**高門檻**連接器，**這決定了它的毛利結構與這次看點**」。事後版要一句話點破真相（例：力積電「EPS 3.36 別當成本業實力」）。
- **常見錯誤**：寫成公司簡介百科；沒有觀點。

### 3.2 近期業績（快報）／法說會實際重點（事後）
- **目的**：釘住最新、已公布的硬數字。
- **必答**：最近 1–2 季營收/EPS/毛利率？年增季增？營收結構占比？（事後）公司對下半年/全年的展望原話？有沒有給正式財測？
- **來源**：TWSE OpenAPI、官方簡報 PDF、財報狗。
- **品質標準**：每個數字標資料日期；**把一次性損益抓出來還原「乾淨」數字**（附錄 A 規則 4）；公司展望用 `✅` 並附「總經理原話」。
- **常見錯誤**：把一次性利益當本業實力；抄新聞成長率不自己核。

### 3.3 估值面
- **目的**：判斷「現在貴不貴」，並灌輸「**好公司 ≠ 好股價**」。
- **必答**：PE/PB 是多少？vs 自身歷史區間？vs 同業？合理價區間？（事後）股價在一年區間的**位置百分位**、近半年漲跌幅？
- **來源**：Yahoo 價、財報狗/Goodinfo 估值。
- **品質標準**：PE 用法人估 EPS 要標 `🔮`；虧損股寫 `PE N/A(虧)`；給合理價區間並和現價比。事後報告一定要算「位置約 X%、近半年 +Y%」判斷風險報酬比是否還漂亮。
- **常見錯誤**：只給一個 PE 數字不比較；不提「已漲一波、追高變貴」。

### 3.4 籌碼面
- **目的**：看聰明錢在做什麼、有無異常。
- **必答**：三大法人買賣超、外資持股、融資融券、是否處置/注意股？「法人 vs 散戶」有無背離？
- **品質標準**：當日籌碼撈不到就標「查無」，不要編。
- **常見錯誤**：把撈不到的數字用印象填。

### 3.5 催化劑時間軸
- **目的**：把「未來會發生什麼」排成時間線。
- **必答**：接下來的關鍵日期——法說/財報/新品/認證/放量/量產。分「近／中」期。
- **品質標準**：多是預判，整段標 `🔮`。

### 3.6 同業比較表
- **目的**：把這檔放進「一組對照」裡看相對位置。
- **必答**：同族群一張表比什麼（股價/PE/PB/EPS 成長/毛利/題材純度/技術路線）？
- **品質標準**：表格**後面必附一句 `> 🔮 解讀：`**，點出這組在賭什麼「分歧」（例：銅 vs 光、買夢 vs 買現在的獲利）。表格只是骨、解讀才是肉。
- **常見錯誤**：只列表不解讀；挑不對的對照組。

### 3.7 財務品質紅旗
- **目的**：找出「數字漂亮但有詐」的地方。
- **必答**：現金流 vs 獲利有無背離？應收/存貨異常？一次性損益佔比多少？
- **品質標準**：**具體把一次性金額抓出來**還原本業（力積電、台達電範例）。這是整份報告最能展現「會質疑」的地方。

### 3.8 多空評估（含 Bear case）
- **目的**：兩面都講，強制對抗偏多。
- **必答**：多方論點？**站在放空立場、最強的看空理由（🐻 空方最強論述）**？
- **品質標準**：Bear case 要真的有殺傷力，不是應付。事後版要對照快報預期是否兌現。
- **常見錯誤**：空方寫得敷衍、只是「風險提醒」貼紙。

### 3.9 🔍 深度延伸（每份必含，這是本專案的靈魂）
獨立一塊 `## 🔍 深度延伸（供應鏈地圖／想像情境／非共識）`，三個子段缺一不可：

- **🕸️ 供應鏈地圖**：
  - 判斷這檔在產業鏈的**真實位置**：直供 Tier-1 還是透過 ODM 的 Tier-2？（決定議價權、是否被低估、景氣反轉先被誰轉嫁）
  - **往上游找「二階／三階受惠」**：賣鏟子給它的人（端子沖壓、電鍍、銅材、線材、功率半導體、磁性元件…），這些市場較少連到題材、是隱藏標的。
  - **同機櫃/同訂單的鄰居**：吃同一台 GB200/GB300 的其他公司（川湖、奇鋐、台達…），可**交叉驗證**出貨動能。
- **💭 想像情境（optionality）**：🐂樂觀／😐中性／🐻保守三劇本 ＋ 被低估的 optionality（明講哪塊「市場給的估值權重幾乎是 0」）。
- **🧠 非共識觀點**：市場目前誤解或還沒反應的點（法人 vs 散戶背離、長線技術替代、被低估折價、剛出爐未消化的催化劑）。**每個非共識要附「怎麼追蹤/何時驗證」**，讓洞察變成可持續觀察的清單。

> 目標是提供「市場共識以外」的洞察，不是複述新聞。標竿：`3526-快報-2026-06-27.md`。

### 3.10 預測追蹤（事後報告專屬）
- **目的**：回填「快報當初預測 vs 法說實際」，建立可回驗的紀律。
- **格式**：三欄表 `快報必盯 | 法說實際 | 落點`，落點用 emoji（🟡符合預期／🐂優於／🐻低於）。範例見 `3526-法說會-2026-07-02.md`。
- **配套**：同一批可驗證預測要登錄到 scorecard（見第 5 節）。

### 3.11 後勢預測與觀察點
- **必答**：短期（H2）看什麼、中期看什麼？給**雙路線收尾**：「重視題材者…；重視獲利驗證者…」（附錄 A 規則 17）。
- **品質標準**：給可追蹤的觀察清單，不是空泛結論。
- **🎯 信心評估（2026-07-05 起必做）**：每個可驗證預測旁標「**信心 X/5**（5🟢很可能／4🟢有機會／3🟡一半一半／2🟠偏低／1🔴不太可能）＋一句理由」，並同步登錄 scorecard 的 `confidence` 欄（見 §5.2）。**信心要誠實反映不確定性**（擇時/股價類別動輒給高分＝校準失效），目標是讓「我說越有把握＝實際越常命中」，未來可回驗出哪類判斷可信。

### 3.12 來源彙整 ＋ 資料時間點
- 文末兩塊：①`資料時間點`：一段話總結「哪些是已公布財報、哪些是法人估、哪些是推測、股價是哪天」。②`來源彙整`：每則**具體標題＋日期＋連結**，不是裸網址。
- 最後一行固定免責：`> 本報告為 AI 輔助研究，僅供參考，不構成投資建議。`
- 若 Andy 說「先別放來源」→ 不寫來源段，只在對話呈現。

## 5. 預測登錄與回驗（scorecard）

### 5.1 檔案位置與用途
- 檔案：`web/src/data/scorecard.json`（若不存在，第一次登錄時建立；建立後前端「預測命中率成績單」讀此檔渲染）。
- 目的：把散在各報告的可驗證預測集中，之後能算「命中率成績單」，讓 Andy 知道這套研究到底準不準。

### 5.2 Schema（設計原則：簡單到人工可維護）
```json
{
  "updated": "2026-07-04",
  "predictions": [
    {
      "pid": "3526-2026-06-27-01",
      "stock_id": "3526",
      "stock_name": "凡甲",
      "report": "3526-快報-2026-06-27.md",
      "predicted_on": "2026-06-27",
      "claim": "GB 系列由試產轉量產、下半年到 2027 逐步放量帶動營收；毛利率守 50%",
      "verifiable": {
        "direction": "up",
        "price_range": null,
        "deadline": "2026-12-31"
      },
      "category": "量產時程",
      "industry": "電子零組件業",
      "confidence": 4,
      "status": "pending",
      "verified_on": null,
      "evidence": null
    }
  ]
}
```

**欄位定義**：

| 欄位 | 型別 | 說明 |
|---|---|---|
| `pid` | string | 唯一鍵，格式 `<stock_id>-<predicted_on>-<兩位序號>` |
| `stock_id` | string | 股票代號 |
| `stock_name` | string | 股票名稱 |
| `report` | string | 來源報告檔名（`web/reports/` 下） |
| `predicted_on` | string | 預測作出日 `YYYY-MM-DD` |
| `claim` | string | 白話預測內容（人看得懂的一句話） |
| `verifiable.direction` | enum | 方向：`up`／`down`／`flat`／`na`（無方向性預測填 na） |
| `verifiable.price_range` | `[low, high]`／`null` | 價位區間預測，沒有就 null |
| `verifiable.deadline` | string | 此預測「該兌現」的期限 `YYYY-MM-DD`（回驗看是否到期用） |
| `category` | enum | 預測類型（分維度統計用）：`量產時程`／`獲利能力`（毛利率/EPS/營收）／`題材發酵`（合作案/認證/新品/訂單）／`股價位置`（目標價/區間）／`產業趨勢`（供需/景氣/替代）。挑最主要的一個 |
| `industry` | string | 該股產業別（照 latest.json／MOPS 的產業欄用詞，例「半導體業」「電子零組件業」；查無填 `""`） |
| `confidence` | int 1–5／null | **分析師對「此預測會兌現」的可能性評估**（calibration 校準用）：`5`🟢很可能(~80%↑)／`4`🟢有機會(~60–80%)／`3`🟡一半一半(~40–60%)／`2`🟠偏低(~20–40%)／`1`🔴不太可能(~<20%)。**每個新預測必填**（2026-07-05 起導入）；此日之前的舊預測填 `null`（不追溯，避免事後諸葛汙染校準）。**信心要誠實反映不確定性**——擇時/股價類預測本質難準，別每筆都給高分，否則校準失去意義。報告內文也要在該預測旁標同一信心＋一句理由。 |
| `status` | enum | `hit`（命中✅）／`miss`（未中❌）／`pending`（未到期⏳） |
| `verified_on` | string／null | 回驗日期 |
| `evidence` | string／null | 回驗依據（實際數字＋來源＋日期） |

> 設計取捨：只用一個扁平陣列 ＋ 每筆一個物件，欄位固定、無巢狀清單（除 `verifiable` 一層），Andy 或任何模型都能手改。`direction/price_range/deadline` 三選項對齊 README「方向／價位區間／時點」三種可驗證條件。

### 5.3 回驗規則
- **寫新報告時順手回驗**：登錄新預測前，先把同一檔股票 `status: pending` 的舊預測過一遍，到期或已可判定的更新 `status`、填 `verified_on` 與 `evidence`。
- **每月至少一次全面回驗**：把所有 `pending` 且 `deadline` 已過的預測結案。
- 回驗原則同最高原則：**準確第一**，`evidence` 要有數字、來源、日期，判不準就維持 `pending` 並註記。
- 每次更新後改 `updated` 欄位為當日（台北時間）。

## 7. 調度規範（主對話／指揮者必讀——多檔研究與上架怎麼跑）

> 觸發語：Andy 說「研究 XXXX」「台股研究 XXXX」（代號或公司名，一次可多檔）＝產快報；「詳細 XXXX」＝產事後報告。公司名先解析成代號再動工。

1. **模型選擇**：研究報告的靈魂是非共識洞察與判斷品質——每檔派一個 **Opus（或當時可用的最高階模型）** subagent 執行，不要用小模型省成本。
2. **並行與隔離**：多檔同時研究就並行派工，**每個 agent 只准寫自己的 `web/reports/<id>-<類型>-<日期>.md`**；scorecard.json、picks.json 是共用檔，agent 一律只讀不寫（並行寫會互相蓋掉）。
3. **agent 的派工 prompt 必含**：①強制第一步＝完整讀本檔並照第 6 節冷啟動，且**明列 §2 資料蒐集鐵律**（禁止裸查無要附查詢軌跡、官方影像簡報要用 pymupdf render 成 PNG 再 Read、資料源打卡、數字用原始資料重算不沿用舊值） ②該股特定事實（上市 .TW／上櫃 .TWO、既有舊報告路徑、要延續檢驗的既有非共識線索）③同檔有舊預測時要求回驗並在回報中給 hit/miss/pending＋依據（不動 JSON）④回報格式：檔案路徑／一句話總結／新預測清單（claim・direction・price_range・deadline・category・**confidence 1-5＋一句理由**）／回驗結果。⑤報告內文每個預測旁要標信心＋理由（見 §3.11、§5.2）。
4. **主對話統一後處理**（agents 全部回來後）：把新預測登錄進 scorecard.json（pid 序號接續）、套用回驗結果（status/verified_on/evidence）、更新 `updated` 欄；有對應近期事件的股票才 add_picks；一次 git commit＋push（push main 自動部署上架）。
5. **交付**：部署完成後驗證線上可開（`r/<檔名去掉.md>/`，例 `…/tw-earnings-calendar/r/3526-快報-2026-06-27/`；2026-07-07 起改真實路徑，發給 Andy 的連結一律用此格式才有正確分享預覽卡，舊 `#/r/` 連結僅自動轉址用），把各報告連結＋一句話總結用 reply 發給 Andy。
6. **品質守門**：主對話抽查每份報告開頭（標記圖例聲明、一句話有沒有觀點、資料時間點段在不在）；不合格退回該 agent 重寫，不要自己補寫。

## 附錄 A：從範例報告歸納的隱性品質規則

> 這些是範例報告展現、但 README 沒明文寫的規則。**這是本檔最有價值的部分**——它們是把報告從「合格」拉到「標竿」的關鍵。

1. **「一句話定位」必帶觀點與「這決定了什麼」。** 不是公司簡介，要點出它為何是這次事件的看點（凡甲：「這決定了它的毛利結構與這次法說會的看點」）。

2. **每個數字後面接「解讀」，用 `> 🔎 解讀：` 引言塊。** 裸數字禁止出現；數字要被翻成投資意義（力積電：淨利暴增 2276% → 解讀「是賣廠帳面利益，不是本業變強」）。

3. **「content per server／單機含量」提問框架。** 不要停在「它是不是 X 的供應鏈」，要逼問「每台機櫃用多少料號、單價多少、含量是否倍增」——這才能解釋毛利質變。反覆出現於凡甲、台達電。

4. **一次性 vs 本業必須分離還原。** 把一次性損益金額抓出來、算出「乾淨」數字（力積電 EPS 3.36 → 本業 0.13；台達電毛利含一次性賠償 0.7%）。這是財務品質紅旗的核心操作，也是「會質疑」最有力的展現。

5. **Tier-1 vs Tier-2 定位法。** 供應鏈地圖必判斷「直供還是透過 ODM」——決定議價權、是否被低估（隱藏版供應鏈）、景氣反轉先被誰轉嫁。

6. **「二階／三階受惠」要列成具體清單**，不是抽象講上游：端子沖壓、電鍍（金/錫）、銅材、線材、功率半導體（MOSFET/GaN）、磁性元件…把「賣鏟子的人」點名。

7. **「同機櫃鄰居」交叉驗證法。** 用同吃一台 GB200/GB300 的公司（川湖、奇鋐、台達）互相驗證出貨動能——這是一種 sanity check，不是湊字數。

8. **同業表後必附一句「路線分歧」解讀。** 表格是骨、`> 🔮 解讀：` 是肉，要點出這組在賭什麼分歧（銅 vs 光、買夢 vs 買現在的獲利）。

9. **事後報告用「股價位置百分位」量化貴不貴。** 「一年區間 218~403、位置約 68%、近半年 +41%」→ 判斷「已非低基期、追高風險報酬比不如快報當時」。這是「好公司≠好股價」的具體算法。

10. **「法說符合預期」本身是資訊。** 沒有驚喜＝來驗證不是來驚喜，能解釋「基本面好但股價沒噴」；管理層「務實 vs 樂觀」的口吻本身是訊號，要點出來。

11. **預測追蹤用三欄表 ＋ emoji 落點**（快報必盯 | 法說實際 | 落點；🟡符合/🐂優於/🐻低於），並在表後一句總結解讀。

12. **誠實聲明放最前面、獨立成塊。** 查無官方材料時，開頭用 `## ⚠️ 資料誠實聲明（請先讀）`，說明用什麼替代、待何更新（訊芯範例）。

13. **「重要更正」機制。** 事後發現先前判斷有誤，開頭用 `## ⚠️ 重要更正（請先讀）` 顯著認錯改正（力積電把「Q1 轉盈來自本業」更正為「來自賣廠一次性」）。禁止偷改。誠實文化＞面子。

14. **非共識觀點要「可追蹤」。** 每個洞察附「怎麼追蹤/何時驗證/何時收斂」，讓它變成持續觀察清單，而非一次性感想。

15. **optionality 用「市場給的估值權重幾乎是 0」句式**明確標出哪塊是純增量、市場還沒計入（凡甲儲能與越南車用直出）。

16. **來源要具體標題＋日期，不是裸連結；文末「資料時間點」獨立總結**哪些是已公布財報、哪些法人估、哪些推測、股價哪天。

17. **雙路線收尾。** 結論給「重視題材者…；重視獲利驗證者…」兩種行動指引，尊重不同風險偏好的散戶，而不是替 Andy 下單一結論。

18. **標記圖例在報告開頭 blockquote 就聲明**（`✅=已驗證｜🔮=推測｜🟡=待確認`），正文逐句貫徹——讓 Andy 一眼看出哪些能信、哪些待查。
